## 14.8 — Ссылка «Админка» в Header

**Цель:** чтобы пользователи с ролью ADMIN или SUPERADMIN видели в шапке сайта явную ссылку «Админка» и могли перейти в админ-панель без ввода URL вручную.

**Текущее состояние:**
- В `components/Header.tsx` уже есть `user` с полем `role` (загружается из `/api/profile`).
- Переменная `cabinetHref` для авторизованных: у ADMIN/SUPERADMIN это `/admin/dashboard`, у ESTABLISHMENT_ADMIN — `/establishment`, у RECIPIENT — `/cabinet`. То есть по кнопке «Кабинет» админ уже попадает в дашборд админки.
- Отдельной ссылки с текстом «Админка» нет; пользователь может не понимать, что «Кабинет» ведёт в админку.

**Что сделать:**

1. **Условие показа:** показывать ссылку «Админка» только если `user?.role === "ADMIN" || user?.role === "SUPERADMIN"`. Для RECIPIENT и ESTABLISHMENT_ADMIN ссылку не показывать.

2. **Места вставки (три зоны в Header):**
   - **Лендинг (desktop):** в блоке `{user ? ( ... )}` рядом с кнопкой «Кабинет» — добавить ссылку «Админка» на `/admin/dashboard` перед «Кабинет» или после (например: «Админка» | «Кабинет» | «Выйти»). Стиль — такой же или чуть приглушённый (например, как вторичная кнопка), чтобы не путать с основным CTA.
   - **Лендинг (мобильное меню):** в сайдбаре в блоке `{user ? ( ... )}` добавить ссылку «Админка» на `/admin/dashboard` перед кнопкой «Кабинет» (или после), с `onClick={close}`.
   - **Не-лендинг (остальные страницы):** в десктопном блоке `{user ? ( ... )}` сейчас только кнопка «Выйти». Добавить перед ней ссылку «Кабинет» (href = `cabinetHref`) и, для ADMIN/SUPERADMIN, ссылку «Админка» (href = `/admin/dashboard`). В мобильном выезжающем меню — аналогично: в блоке с пользователем добавить «Кабинет» и «Админка» (последняя только для ADMIN/SUPERADMIN).

3. **Текст и доступность:** текст ссылки — «Админка»; для иконки можно опционально использовать что-то вроде Shield или Settings из lucide-react. `aria-label="Перейти в админ-панель"` при необходимости.

4. **Активное состояние (опционально):** если `pathname.startsWith("/admin")`, можно подсветить ссылку «Админка» (другой класс), чтобы было понятно, что пользователь в админке.

**Итог:** правки только в `components/Header.tsx`, без новых API и без изменений в бэкенде. Оценка: 15–30 минут.

## 12.2 — OpenAPI (Swagger) для API

**Цель:** иметь машиночитаемое описание API (OpenAPI 3.x) и при желании интерактивную документацию (Swagger UI), чтобы фронтенд и мобильное приложение могли по контракту генерировать клиенты или сверять запросы/ответы.

**Текущее состояние:**
- Документация в виде Markdown: `docs/API.md` с описанием auth, profile, links, transactions, payouts, admin, pay, webhook.
- Эндпоинтов много (десятки); описание вручную в MD не синхронизировано автоматически с кодом.

**Что сделать:**

1. **Выбор подхода:**
   - **Вариант A:** Ручная/полуавтоматическая спецификация — один или несколько YAML/JSON файлов OpenAPI 3.x (например `openapi.yaml` в корне или в `docs/`), которые заполняются вручную или скриптом по шаблону. Эндпоинты, параметры, тела запросов и примеры ответов описываются вручную на основе `docs/API.md` и реальных route-файлов.
   - **Вариант B:** Генерация из кода — использовать библиотеку (например для Next.js route handlers — нет единого стандарта; возможны обходы через JSDoc/декораторы или отдельный слой, который импортирует хендлеры и извлекает метаданные). Часто требует дисциплины в коде (единый формат ответов, аннотации).
   - **Рекомендация для быстрого результата:** Вариант A — один `openapi.yaml` с основными группами: auth (register, login, refresh, logout, forgot-password, reset-password), profile (GET/PATCH, change-password), links (GET, POST), transactions (GET), payouts (GET, POST), pay (GET/POST по slug), admin (stats, payouts, users) и webhook. Секция `securitySchemes`: Bearer (JWT), при необходимости cookie для refresh. Сервер в spec — переменная (например `url: ${NEXT_PUBLIC_APP_URL}/api` или условно для локальной разработки).

2. **Структура спецификации:**
   - `openapi: 3.0.x` (или 3.1).
   - `info`: title (FreeTips / 1tips), version (из package.json или фиксированный), description — кратко.
   - `servers`: один или два (local, production).
   - `paths`: для каждого пути — метод, summary, tags (Auth, Profile, Links, …), security (где нужен Bearer), requestBody (где есть), parameters (query, path), responses (200, 400, 401, 403, 429, 503) с schema или example.
   - `components/schemas`: переиспользуемые схемы (User, TipLink, Transaction, PayoutRequest, ErrorResponse и т.д.).
   - `components/securitySchemes`: Bearer JWT.

3. **Swagger UI (интерактивная документация):**
   - Либо отдельный маршрут в приложении, например `app/api-docs/page.tsx` (или `app/docs/api/page.tsx`), который рендерит Swagger UI (пакет `swagger-ui-react`), подгружая `openapi.yaml` с публичного URL или из статики (`/openapi.yaml` в `public/`).
   - Либо отдельная HTML-страница в `public/` (swagger-ui standalone), которая тянет spec по URL. Минус — нужен доступ к spec по тому же origin или CORS.
   - Учитывать, что часть эндпоинтов защищена: в Swagger UI можно добавить кнопку «Authorize» и ввод Bearer token (accessToken), чтобы пробовать запросы из браузера.

4. **Обслуживание:** при добавлении/изменении эндпоинтов обновлять `openapi.yaml` (и при необходимости `docs/API.md`). В README или CONTRIBUTING указать, что контракт API описан в OpenAPI и что перед релизом стоит проверить актуальность.

**Итог:** новый файл спецификации OpenAPI, опционально страница с Swagger UI и упоминание в документации. Оценка: 4–8 часов на первый полный проход по основным эндпоинтам; дальше — точечные правки.

## 11.1 и 11.2 — Юнит- и интеграционные тесты

**Цель:** повысить надёжность и упростить рефакторинг за счёт автоматических тестов для критичных сценариев: авторизация, баланс, idempotency (11.1) и сценарии запросов к API (11.2).

**Текущее состояние:**
- В `package.json`: `"test": "tsx --test tests/validation/*.test.ts tests/security/*.test.ts tests/integration/*.test.ts"`.
- Есть тесты: `tests/validation/login.test.ts`, `tests/validation/payment.test.ts`, `tests/security/rate-limit.test.ts`, `tests/security/csrf-client.test.ts`, `tests/integration/health.test.ts`.
- Node.js test runner (`node:test`) + `tsx`; покрытие через `c8` (скрипт `test:coverage`).

**11.1 — Юнит-тесты (сервисы auth, баланс, idempotency):**

- **Auth (логика без HTTP):**
  - Функции из `lib/auth/` (например хеширование пароля, генерация/проверка JWT, генерация/хеш токена сброса пароля) — тестировать изолированно: хеш пароля не совпадает с исходным; два хеша одного пароля различаются по соли; JWT создаётся и верифицируется с одним секретом, падает с другим; срок жизни токена учитывается.
  - Файлы: например `tests/unit/auth/password.test.ts`, `tests/unit/auth/jwt.test.ts`, `tests/unit/auth/password-reset-token.test.ts`. Импорт функций из `lib/auth/` и вызов с разными аргументами, проверка через `assert`.
- **Баланс:**
  - Логика расчёта баланса (например в `lib/balance.ts` или там, где считается «сумма успешных транзакций минус успешные выводы») — если она вынесена в отдельные функции, написать тесты на граничные случаи: пустой список транзакций/выводов, только успешные, смесь статусов (PENDING, SUCCESS, FAILED), большие числа (BigInt). Если логика только внутри route handler и сильно завязана на Prisma, можно покрыть через интеграционные тесты (11.2) с моком БД или тестовой БД.
- **Idempotency:**
  - Если где-то есть idempotency key (например при создании платежа/транзакции) — юнит-тест: два запроса с одним ключом возвращают один и тот же результат и не создают дубликата; разные ключи создают две сущности. Место реализации уточнить по коду (например `app/api/pay/[slug]/route.ts` или слой создания транзакции).

**11.2 — Интеграционные тесты (POST /auth/*, GET /profile, GET /transactions):**

- **Подход:** вызывать реальные route handlers (как в `health.test.ts`): `import { POST } from "../../app/api/auth/login/route"` и т.д., передавать `NextRequest` (можно сформировать через `new Request(url, { method, headers, body })`). БД: либо тестовая PostgreSQL (docker compose с отдельной БД для тестов), либо мок Prisma/`db` на уровне модуля (сложнее и хрупче).
- **Сценарии:**
  - **POST /api/auth/register:** валидный body → 201, в ответе есть accessToken и user; невалидный (дубликат логина) → 409; невалидные поля (короткий пароль, пароли не совпадают) → 400.
  - **POST /api/auth/login:** верные логин/пароль → 200, есть accessToken; неверный пароль → 401; несуществующий логин → 401.
  - **GET /api/profile:** без токена → 401; с валидным Bearer (создать пользователя через register или напрямую в БД, затем получить токен через login) → 200, в теле login/email/role.
  - **GET /api/transactions:** без токена → 401; с токеном и пустым списком транзакций → 200, массив пустой или поле с массивом пустое; при наличии транзакций — проверка структуры элемента (id, amountKop, status, createdAt и т.д.).
- **Дополнительно (по желанию):** POST /api/auth/refresh (с cookie), POST /api/auth/logout, GET /api/links — по той же схеме.
- **Организация:** один файл `tests/integration/auth-api.test.ts` или разбить на `auth.test.ts`, `profile.test.ts`, `transactions.test.ts`. В начале тестов при использовании реальной БД — создание тестового пользователя и при необходимости очистка после теста (или использование отдельной схемы/БД).

**Итог:** новые файлы в `tests/unit/` и `tests/integration/`, расширение `npm run test`. Оценка: 11.1 — 2–4 часа, 11.2 — 3–5 часов в зависимости от того, поднимается ли тестовая БД или используется мок.

## 12.4 — CI (линт, тесты, сборка на каждый push)

**Цель:** на каждый push (и/или PR) в репозиторий автоматически запускать линт, тесты и сборку приложения, чтобы ловить поломки до продакшена.

**Что сделать:**

1. **Платформа:** GitHub Actions (если репозиторий на GitHub). Для GitLab — аналог в `.gitlab-ci.yml`; для других хостов — свой CI-конфиг.

2. **Файл workflow:** создать `.github/workflows/ci.yml` (или `main.yml`):
   - **Триггеры:** `push` в ветки (например `main`, `develop`) и `pull_request` в те же ветки.
   - **Джоба (одна или несколько):**
     - **Checkout:** `actions/checkout@v4`.
     - **Node:** установка Node.js (версия из `.nvmrc` или явно, например 20).
     - **Кэш зависимостей:** `actions/cache` по ключу `node-modules-{{ hashFiles('package-lock.json') }}` (или yarn/pnpm) для папки `node_modules` и шага `npm ci` (или `yarn install --frozen-lockfile`), чтобы ускорить последующие запуски.
     - **Шаги по порядку:**
       1. `npm ci` (или аналог) — установка зависимостей.
       2. `npm run lint` — ESLint. При падении линта — fail джобы.
       3. `npm run test` — юнит и интеграционные тесты. Если тесты требуют БД или Redis, добавить сервис-контейнер (PostgreSQL, Redis) в job или вынести тесты, требующие БД, в отдельный job с `services: postgres: ...`.
       4. `npm run build` — сборка Next.js (включает `prisma generate`). Убедиться, что в CI заданы обязательные env (например `DATABASE_URL` заглушка или реальная тестовая БД; `JWT_SECRET`, `JWT_REFRESH_SECRET` — можно сгенерировать одноразовые для сборки, если приложение не подключается к БД на этапе build кроме generate).

3. **Переменные окружения для CI:**
   - В репозитории (Settings → Secrets and variables → Actions) при необходимости задать секреты; для сборки часто достаточно задать в `ci.yml` фиктивные значения: `JWT_SECRET`, `JWT_REFRESH_SECRET`, `DATABASE_URL=postgresql://...` на сервисный контейнер postgres. Если в `next build` есть обращение к БД — либо мокать, либо поднять postgres в CI.

4. **Итоговая схема:** один workflow с одной job (или две: lint + test без БД, затем build + integration tests с БД). На каждом push/PR будет виден статус: passed / failed. В README можно добавить бейдж «CI» (например GitHub Actions badge).

**Итог:** один YAML-файл в `.github/workflows/`, при необходимости — настройка сервисных контейнеров и переменных. Оценка: 1–2 часа с учётом отладки под текущий `npm run build` и тесты.

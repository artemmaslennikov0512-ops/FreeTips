# План: гибкое редактирование кабинета официанта и QR для заведения

## Цели

1. **Гибкое редактирование кабинета официанта из ЛК управляющего** — управляющий может менять данные сотрудника (имя, должность, коэффициент, активность) и при необходимости ограничивать видимость разделов кабинета официанта.
2. **QR для заведения** — создание, редактирование макета и скачивание файла для печати (один QR заведения и/или листы с QR сотрудников).

---

## Часть 1. Гибкое редактирование кабинета официанта из ЛК управляющего

### Текущее состояние

- **Роли:** `ESTABLISHMENT_ADMIN` (управляющий), `EMPLOYEE` (официант).
- **Страница «Команда»:** `/establishment/team` — список сотрудников, добавление, деактивация, выдача ссылки/инвайт, скачивание PDF с QR по сотрудникам.
- **API редактирования:** `PATCH /api/establishment/employees/[id]` — уже есть поля: `isActive`, `name`, `position`, `coefficient`. Редактирования имени/должности/коэффициента из UI команды **нет** (только деактивация).

### Что сделать

| № | Задача | Детали |
|---|--------|--------|
| 1.1 | **Редактирование карточки сотрудника в «Команде»** | На странице `/establishment/team`: кнопка «Редактировать» у каждого сотрудника → модалка/форма с полями: имя, должность, коэффициент. Сохранение через существующий `PATCH /api/establishment/employees/[id]`. Валидация на бэкенде уже есть. |
| 1.2 | **Ограничение видимости разделов кабинета официанта (опционально)** | Если нужно, чтобы управляющий мог включать/выключать разделы ЛК официанта (например «Моя ссылка», «Операции», «Верификация»): добавить в `Establishment` или в `Employee` флаги/настройки (например JSON `cabinetVisibleSections: string[]` или отдельные поля). В `app/cabinet/layout.tsx` и на страницах кабинета — проверка этих настроек и скрытие пунктов меню/редирект. API: `PATCH /api/establishment/settings` или отдельный эндпоинт для «настроек кабинета сотрудников». |
| 1.3 | **Единая страница сотрудника (опционально)** | Отдельный маршрут `/establishment/team/[id]` — просмотр и редактирование одного официанта, статистика, ссылка, инвайт, история — для удобства при большом числе сотрудников. |

**Рекомендация:** начать с **1.1** (редактирование в таблице/карточке). Пункты 1.2 и 1.3 — по желанию продукта.

---

## Часть 2. QR для заведения: создание, редактирование, скачивание для печати

### Текущее состояние

- Есть **QR по сотрудникам:** ссылка вида `/pay/{qrCodeIdentifier}` (slug сотрудника). PDF с карточками генерируется в `GET /api/establishment/employees/pdf` — фиксированный макет (2 колонки, 4 ряда, имя, должность, QR, URL).
- У заведения есть **uniqueSlug** в БД, но отдельной страницы «оплата по заведению» (один QR на всё заведение) сейчас нет.

### Что сделать

| № | Задача | Детали |
|---|--------|--------|
| 2.1 | **QR «на заведение» (одна ссылка на весь ресторан)** | Вариант A: страница `/pay/establishment/[slug]` или `/e/[slug]` по `Establishment.uniqueSlug`: гость переходит по одному QR, видит список официантов (или «Чаевые в общий пул») и выбирает, кому оставить чаевые. Вариант B: один QR ведёт сразу в пул заведения (без выбора). Реализация: новый маршрут в `app/pay/...`, API `GET /api/pay/establishment/[slug]` (инфо о заведении + список активных сотрудников при варианте A). TipLink для заведения: либо один общий TipLink с slug = uniqueSlug (или отдельное поле), либо редирект на выбор. |
| 2.2 | **Страница «QR и печать» в ЛК управляющего** | Новая страница `/establishment/qr` (или подраздел в «Команда» / «Настройки»): блок «QR заведения» (ссылка + превью QR + скачать PNG/SVG), блок «Карточки сотрудников для печати» — параметры макета и кнопка «Скачать PDF». |
| 2.3 | **Редактирование макета PDF для печати** | Параметры (сохранять в localStorage или в настройках заведения в БД): размер карточки, кол-во карточек на странице, показывать/скрывать текст ссылки под QR, логотип заведения на карточке, подпись (например «Чаевые для [Имя]»). API: `GET /api/establishment/employees/pdf?cardWidth=...&cardHeight=...&showUrl=...` с query-параметрами или тело запроса для сложных настроек. Либо сохранять пресет в `Establishment` (например JSON `qrPdfSettings`). |
| 2.4 | **Скачивание файла для печати** | Уже есть: PDF по сотрудникам. Добавить: скачивание **одного QR заведения** в виде PNG/SVG для вставки в визитку/стикер (если делаем 2.1). Формат: ссылка на `/pay/establishment/{uniqueSlug}` или на страницу выбора официантов. |

**Рекомендация:** 2.2 и 2.3 дают «гибкое редактирование и скачивание файла для печати»; 2.1 и 2.4 — ценное дополнение «один QR на заведение».

## Часть 3. Дополнения от разработки

| № | Предложение | Зачем |
|---|-------------|--------|
| 3.1 | **Валидация и лимиты** | При редактировании коэффициента/имени — показывать ошибки валидации под полями (сейчас API возвращает сообщение — вывести его в форме). Напоминать про лимит `maxEmployeesCount` при добавлении. |
| 3.2 | **Аудит изменений (логи)** | Логировать в приложении (или в БД) ключевые действия управляющего: редактирование сотрудника, деактивация, выдача инвайта — для разбора спорных ситуаций (без перегрузки первой итерации). |
| 3.3 | **Превью PDF перед скачиванием** | На странице QR/печати: кнопка «Предпросмотр» — открыть PDF в новой вкладке; «Скачать» — как сейчас. Удобно проверить макет перед печатью. |
| 3.4 | **Единый стиль карточек и заведения** | В PDF использовать цвета/логотип заведения из раздела «Бренд» (уже частично есть primaryColor). Расширить использование `logoUrl`, `secondaryColor` в макете карточки. |

## Порядок внедрения (приоритеты)

1. **Фаза 1 (минимум)**  
   - 1.1 — редактирование сотрудника (имя, должность, коэффициент) со страницы «Команда».  
   - 2.2 — страница «QR и печать» с текущим PDF и кнопкой скачать; при необходимости вынести кнопку «Скачать PDF» сюда.  
   - 2.3 (базово) — 1–2 параметра макета PDF (например размер карточки или показ/скрытие URL) через query-параметры API.

2. **Фаза 2**  
   - 2.1 — QR на заведение (страница выбора официанта или переход в пул).  
   - 2.4 — скачивание одного QR заведения (PNG/SVG).  
   - 2.3 (полно) — сохранение пресетов макета в настройках заведения.

3. **Фаза 3 (по желанию)**  
   - 1.2 — настройка видимости разделов кабинета официанта.  
   - 1.3 — отдельная страница сотрудника `/establishment/team/[id]`.  
   - 3.1–3.4 — валидация в формах, превью PDF, аудит, единый стиль бренда в PDF.

## Технические точки

- **Права:** все новые эндпоинты для управляющего — `requireEstablishmentAdmin`, проверка `establishmentId`.
- **Публичные маршруты:** `/pay/establishment/[slug]` и API для него — без авторизации, только проверка существования заведения и активных сотрудников.
- **Файлы:** генерация PDF на сервере (как сейчас); генерация PNG/SVG QR — на сервере (например `qrcode` в data URL и отдача файла) или на клиенте и скачивание через blob.
- **БД:** при необходимости настроек кабинета/пресетов PDF — миграции Prisma для новых полей в `Establishment`.

После утверждения плана можно переходить к задачам по фазам (начиная с 1.1 и 2.2).

# Сценарий работы с заведением

**Проект:** 1tips  
**Связь:** [IMPLEMENTATION_PLAN_ESTABLISHMENTS.md](./IMPLEMENTATION_PLAN_ESTABLISHMENTS.md), [ANALYSIS_ESTABLISHMENT_EMPLOYEE_SPEC.md](./ANALYSIS_ESTABLISHMENT_EMPLOYEE_SPEC.md)

Ниже — сценарии для разных ролей: что делают суперадмин, админ заведения, официант и гость. Сначала текущее состояние, затем целевое после внедрения заведений.

---

## 1. Текущее состояние (без модели Establishment)

| Роль | Сценарий |
|------|----------|
| **Суперадмин** | Одобряет заявки на регистрацию; в заявке указано текстовое поле «Заведение». В админке видит пользователей и их заведение (строка). |
| **Пользователь (RECIPIENT)** | Регистрируется по токену, в профиле может указать название заведения (текст). Имеет ссылки для чаевых (TipLink), выводы. |
| **Гость** | Переходит по ссылке вида `/pay/{slug}`, оплачивает чаевые — средства зачисляются получателю по slug. |

Ограничение: заведение — только подпись (строка), нет сущности «заведение», нет админа заведения и распределения чаевых по команде.

---

## 2. Целевые сценарии (после внедрения заведений)

### 2.1 Суперадмин

1. **Создание/одобрение заведения**
   - Вариант A: заведение создаётся при одобрении заявки на регистрацию (если в заявке указано новое название — создать запись Establishment и выдать пользователю роль ESTABLISHMENT_ADMIN и привязку к этому заведению).
   - Вариант B: в суперадминке отдельный раздел «Заведения» → «Добавить заведение» (название, адрес, телефон, uniqueSlug) → затем привязка пользователя как админа этого заведения (через редактирование пользователя: роль + establishmentId).

2. **Контроль**
   - Просмотр списка заведений, списка пользователей с фильтром по заведению.
   - Блокировка пользователя (isBlocked) или при необходимости блокировка заведения (если введём флаг isActive у Establishment).

3. **Выплаты**
   - Как сейчас: заявки на вывод обрабатываются вручную; опционально — фильтр по заведению для отчётов.

---

### 2.2 Админ заведения (ESTABLISHMENT_ADMIN)

1. **Вход и доступ**
   - Вход в личный кабинет; в меню пункт «Админ заведения» → переход в раздел `/establishment` (доступ только при role === ESTABLISHMENT_ADMIN и заполненном establishmentId).

2. **Управление командой**
   - **Команда** (`/establishment/team`): список сотрудников (Employee) своего заведения.
   - **Добавить сотрудника**: имя, должность (опционально), коэффициент (для распределения). Система генерирует уникальный slug (qrCodeIdentifier) и при необходимости создаёт TipLink для приёма оплат по этому slug.
   - **QR и PDF**: просмотр ссылки для чаевых по каждому сотруднику; кнопка «Скачать PDF с QR-кодами» для выбранных сотрудников.
   - **Деактивация**: снять галочку «Активен» у сотрудника (уволен/в архиве) — чаевые по его QR дальше не учитываются в распределении или не принимаются (по правилу продукта).

3. **Распределение чаевых**
   - Страница «Распределение» (`/establishment/payout-rules`): настройка правил (равномерно / по коэффициентам / гибрид).
   - При необходимости: ручной ввод «общих» чаевых за смену/день (наличные, терминал) → сумма попадает в пул и распределяется по правилам.
   - Запуск расчёта распределения за период → просмотр результата (кто сколько получил); подтверждение → начисление на балансы привязанных пользователей (или запись в Allocation). История распределений только для чтения (аудит).

4. **Аналитика**
   - Отчёты по заведению: сумма чаевых за период, по сотрудникам, по дням.
   - Экспорт CSV/PDF для бухгалтерии.

---

### 2.3 Официант (EMPLOYEE)

1. **Привязка к заведению**
   - Админ заведения создаёт сотрудника и либо приглашает по email (ссылка на регистрацию/привязку), либо вручную привязывает существующий аккаунт к записи Employee (userId). После привязки у пользователя role = EMPLOYEE и связь с заведением через Employee.

2. **Личный кабинет**
   - Видит баланс, историю транзакций (чаевые по своему QR).
   - Блок «Начисления от заведения»: суммы, пришедшие из распределения общих чаевых за смену/период (после реализации Фазы 3).
   - Опционально: рейтинг, история отзывов (если продукт введёт отзывы при оплате).

3. **Вывод**
   - Как у обычного получателя: заявка на вывод → обработка суперадмином (или автоподтверждение по правилам). Для сотрудников заведения может применяться пониженная комиссия (конфиг).

---

### 2.4 Гость (клиент заведения)

1. **Оплата чаевых**
   - Сканирует QR официанта или переходит по ссылке вида `https://.../pay/{slug}` (slug = qrCodeIdentifier сотрудника или общая ссылка заведения).
   - На странице оплаты отображаются лого/название заведения (White Label, если включено).
   - Выбирает способ оплаты (карта, Apple Pay и т.д.), вводит сумму → оплата.
   - Деньги зачисляются: при личном QR — на счёт официанта (или в пул заведения с последующим распределением, в зависимости от настроек); при «общей» ссылке заведения — в пул заведения.

2. **Опционально**
   - После оплаты — короткий отзыв/рейтинг по официанту (геймификация, если введена в продукт).

---

## 3. Краткая последовательность (жизненный цикл заведения)

```
1. Суперадмин создаёт заведение (или оно создаётся при одобрении заявки).
2. Суперадмин назначает пользователя админом этого заведения (роль ESTABLISHMENT_ADMIN, establishmentId).
3. Админ заведения добавляет сотрудников (Employee), получает QR/PDF.
4. Официанты привязываются к записям Employee (регистрация по инвайту или ручная привязка).
5. Гости платят по QR/ссылке → чаевые зачисляются (личные или в пул).
6. Админ настраивает правила распределения и при необходимости вносит «общие» чаевые, запускает распределение.
7. Официанты видят начисления и выводят средства.
8. Суперадмин контролирует заведения и выплаты; при необходимости деактивирует сотрудника или блокирует пользователя.
```

---

## 4. Что реализовать по шагам

| Шаг | Действие | Результат |
|-----|----------|-----------|
| Фаза 1 | Модели Establishment, Employee, PayoutRule; роли ESTABLISHMENT_ADMIN, EMPLOYEE; User.establishmentId | Данные и права готовы, старый поток не ломается |
| Фаза 2 | Раздел `/establishment`, команда (CRUD сотрудников, QR, PDF), настройка правил распределения | Админ заведения управляет командой и правилами |
| Фаза 3 | Связь slug ↔ Employee, пул общих чаевых, сервис распределения, аудит, отчёты | Полный цикл чаевых и распределения по заведению |
| Фаза 4 | Кабинет официанта (начисления), комиссии для заведений | Удобный UX для официанта и гибкие комиссии |
| Фаза 5 | White Label, юридика/ФНС, опционально благотворительность/геймификация | Брендинг и соответствие требованиям |

Подробные задачи по фазам — в [IMPLEMENTATION_PLAN_ESTABLISHMENTS.md](./IMPLEMENTATION_PLAN_ESTABLISHMENTS.md).
